# AGENTS.md

## 项目名称
水下目标检测网页系统（Underwater Target Detection Web System）

## 项目目标
构建一个网页端水下目标检测系统，支持：
1. 图片检测
2. 视频检测
3. 视频统计分析
4. 结果展示与下载

当前已具备条件：
- 已训练好的 PyTorch `.pt` 权重文件（本地）
- 项目托管在 GitHub
- 后续主要通过 Cursor 的 Codex 插件开发
- 当前阶段优先完成可用的第一版系统

---

## 技术栈约束

### 后端
- Python 3.10
- FastAPI
- PyTorch
- OpenCV
- Uvicorn

### 前端
- Vue 3
- Element Plus
- ECharts
- Vite

### 存储
- 第一阶段不强制数据库
- 使用文件系统 + JSON 保存结果元信息
- 如需历史记录，可后续加入 SQLite

---

## 当前功能范围（MVP）
必须优先实现以下功能：

### 1. 图片检测
- 上传图片
- 调用模型进行推理
- 返回带检测框的图片
- 返回类别、数量、置信度
- 支持结果下载

### 2. 视频检测
- 上传视频
- 逐帧检测
- 生成带检测框视频
- 返回视频分析统计结果
- 支持结果下载

### 3. 视频分析功能
必须包含以下三项：
- 每类目标出现的总帧数
- 按时间轴显示目标出现区间
- 每秒目标数量变化曲线

---

## 功能实现要求

### 图片检测页
页面需要支持：
- 图片上传
- 原图预览
- 检测结果图展示
- 检测目标列表展示
- 类别数量统计
- 置信度展示
- 下载结果图
- 下载检测结果 JSON

### 视频检测页
页面需要支持：
- 视频上传
- 参数设置（至少支持 conf、iou）
- 检测结果视频播放
- 结果下载
- 统计信息展示

### 视频统计展示
必须展示：
1. 每类目标出现总帧数
2. 每类目标出现区间
3. 每秒目标数量变化曲线

推荐展示方式：
- 表格
- 柱状图
- 折线图
- 时间轴或区间列表

## 架构要求
采用模块化设计，禁止将所有逻辑写入单文件。

### 后端建议目录
- `api/`：路由接口
- `services/`：业务逻辑
- `models/`：模型加载与推理
- `utils/`：工具函数
- `schemas/`：请求响应数据结构
- `core/`：配置与基础组件

### 前端建议目录
- `views/`：页面
- `components/`：组件
- `api/`：接口请求封装
- `utils/`：工具函数
- `router/`：路由
- `stores/`：状态管理（若需要）

---

## 编码规则

### 通用
- 代码必须模块化
- 变量命名清晰
- 不允许硬编码本地绝对路径
- 不允许把权重路径写死为某台机器的路径
- 不允许生成无说明的大量样板代码
- 每个文件职责必须明确

### Python
- 尽量添加类型注解
- 核心函数写简洁注释
- 接口返回结构保持统一
- 文件读写路径通过配置项获取

### Vue
- 页面与组件拆分合理
- 不要把全部逻辑堆在一个 `.vue` 文件中
- 图表逻辑独立封装
- 上传、结果展示、统计分析应分模块实现

---

## 配置规则
所有可变配置必须通过配置文件或环境变量控制，例如：
- 模型路径
- 上传目录
- 输出目录
- 元数据目录
- 默认 conf
- 默认 iou
- 端口号

不得在代码中写死：
- `D:/...`
- `C:/Users/...`
- `/home/...`
- 其他个人机器路径

---

## 文件与结果管理规则
项目中应使用以下目录：
- `weights/`：模型权重
- `uploads/`：用户上传文件
- `outputs/`：输出结果文件
- `metadata/`：检测结果 JSON / 历史信息

文件命名应尽量避免冲突，建议使用：
- 时间戳
- UUID
- 原始文件名摘要

---

## 输出要求
在实现功能时，优先输出：
1. 目录结构
2. 文件清单
3. 核心代码
4. 运行方式
5. 后续待完善项

如果任务较大，优先给出最小可运行版本，不要一次生成过于复杂的系统。

## 与本项目相关的重要背景
这是一个“水下目标检测系统”，不是通用管理后台。
设计时应突出：
- 检测可视化
- 视频分析
- 统计展示
- 项目演示效果

在界面和代码组织上，应优先服务于“图片检测 + 视频检测 + 数据分析”。
